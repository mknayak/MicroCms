Executive SummaryThis gap analysis compares the fifteen UI design screens of MicroCMS v2 against the backend code delivered in Sprints 1–5. Sprints 1–5 established the foundational domain model (Site, ContentType, Entry, MediaAsset, Taxonomy), the Infrastructure layer (EF Core, PostgreSQL, Repository, UnitOfWork, Outbox), and the Application layer (MediatR CQRS pipeline, FluentValidation, Mapperly, authorization behaviours, full Entry CRUD with version history and scheduling).While the core content management backbone is solid, the design screens reveal a substantial set of features — spanning AI assistance, page design tooling, component library, media enrichment, webhook infrastructure, and rich tenant/site configuration — that are not yet implemented. Twenty-eight distinct feature gaps have been identified and catalogued below, with priority and effort estimates to guide sprint planning.1.1  What Sprints 1–5 DeliveredThe following areas are implemented and tested:Domain Model: Site, ContentType (field schema stub), Entry (status machine, versioning, scheduling), MediaAsset, Taxonomy, OutboxMessage — all with strongly-typed IDsInfrastructure: EF Core 8 with global multi-tenant query filter, Outbox interceptor, SQLite & PostgreSQL migrations, Repository<T>, UnitOfWork, Testcontainers integration testsApplication CQRS: MediatR 12 pipeline (Logging → Authorization → Validation → UnitOfWork), ICommand/IQuery markers, FluentValidation, Mapperly DTOsEntry Commands: Create, Update, Publish, Unpublish, Delete, SchedulePublish, RollbackToVersionEntry Queries: GetEntry, ListEntries (paged), GetEntryVersionsAuthorization: Role-based policies (SystemAdmin, TenantAdmin, Editor, Author, Viewer), HasPolicyAttribute, fail-secure MissingPolicyExceptionUnit Tests: 50+ tests covering all command/query handlers, validators, authorization behaviours, and role permissions2. Architecture Notes for Gap RemediationThe following principles should guide implementation of the identified gaps, consistent with the existing codebase conventions established in Sprints 1–5.6.1  Domain Model ExtensionsEach new aggregate (Component, Page, WebhookSubscription, ApiClient, etc.) must follow the existing pattern: strongly-typed ID record struct, private field constructor, static Create() factory, domain events via RaiseDomainEvent(), IRepository<T, TId> registration.Value objects (SeoMetadata, SiteEnvironment, AiBudget, SecurityConfig) must be immutable record types in MicroCMS.Domain, placed in the relevant ValueObjects subfolder.FieldDefinition for the typed field system should be a value object in Domain.Aggregates.Content, with a FieldType enum and an optional ValidationRules value object.6.2  Application LayerEvery new command and query must carry [HasPolicy(ContentPolicies.XxxYyy)] — no exceptions. Use the existing fail-secure authorization pattern.AI-related commands (DraftContent, TranslateEntryLocale, GenerateAltText) should declare ICommand<string> or ICommand<AiResultDto> and route through the pipeline in the normal way. The AI provider call belongs in the handler, not a behavior.Bulk commands (BulkPublishEntries, BulkDeleteEntries) should iterate individual domain methods to preserve invariants — do not bypass the Entry state machine.6.3  InfrastructureILlmService abstraction should follow the Adapter pattern: one interface, multiple infrastructure implementations (AnthropicLlmService, AzureOpenAiLlmService, etc.), selected via IOptions<AiProviderSettings> at runtime. This matches the design's ADR-011 'provider swap requires only a configuration change'.ClamAV integration should be an IVirusScanService in Infrastructure, invoked from the UploadMediaAssetCommandHandler, not as middleware. The plugin architecture shown in the design maps to DI plugin assemblies.Webhook dispatch should be triggered by domain events via the Outbox pattern already implemented: a BackgroundService reads OutboxMessages of type 'domain-event' and dispatches to registered WebhookSubscriptions.6.4  Security ConsiderationsPreview tokens: use a short-lived HMAC-SHA256 signed JWT (max 1 hour), never stored in DB — stateless verification on the server using a per-site secret.API client secrets: store only the bcrypt hash, never the plaintext. The raw key is shown exactly once on creation (same pattern as GitHub PATs).Webhook HMAC signing: sign the entire payload with HMAC-SHA256 using the per-site webhook secret. Include X-MicroCMS-Signature and X-MicroCMS-Timestamp headers to prevent replay attacks.AI PII redaction: run a regex-based pre-processor on content before sending to any AI provider. Log redaction events to the audit log without storing the redacted values.6.5  Complexity & Test CoverageCyclomatic complexity ≤ 10 per method (project constraint). Bulk commands should delegate to a private helper that handles a single entry to keep the loop body simple.All new command handlers must have unit tests using NSubstitute + FluentAssertions + xUnit, following the Arrange/Act/Assert pattern established in Sprint 3.AI service integration tests should use recorded cassettes (VCR pattern) or a test double implementing ILlmService to avoid network calls in CI.7.
